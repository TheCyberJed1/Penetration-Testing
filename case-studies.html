Scenario

Simulated an external attacker who gained initial access to an internal network segment (representing a phished employee's laptop on a corporate LAN). Starting from zero domain credentials, the goal was to achieve Domain Admin.

Attack Path Summary

  • Ran Responder to capture NTLMv2 hashes on the network segment
  • Cracked captured hash offline using Hashcat with rockyou.txt wordlist in under 4 minutes
  • Authenticated to SMB share with recovered credentials — found cleartext password in a config file
  • Used BloodHound to identify a Kerberoastable service account with a path to Domain Admins
  • Kerberoasted the service account; cracked the TGS ticket offline
  • Leveraged compromised service account privileges to DCSync and dump all domain hashes

Key Findings

NTLM Hash Capture via Responder Weak Password Policy (8-char, no complexity) Kerberoastable Service Account with DA Path Cleartext Credentials in Network Share Excessive SMB Share Permissions

Remediation Recommendations Documented

  • Enable SMB signing on all domain controllers and workstations
  • Enforce minimum 15-character passwords with complexity requirements
  • Audit and rotate service account passwords; implement Managed Service Accounts (gMSA)
  • Remove cleartext credentials from shared drives; audit share permissions quarterly
  • Implement tiered Active Directory administration model
Key Takeaway: A single weak password on a domain-joined workstation was the initial foothold that led to full domain compromise. Most AD environments have similar exposure — this attack chain is well-documented in the wild and actively exploited by ransomware groups.

Scenario

A small nonprofit organization with a WordPress-based donation and member management portal requested a pro-bono web application security assessment. The site handled donor PII and processed payments through a third-party gateway. Testing was conducted under a signed authorization agreement during off-peak hours.

Testing Approach

  • Authenticated and unauthenticated testing of the WordPress application
  • Enumeration of plugins, themes, and WordPress version using WPScan
  • Manual OWASP Top 10 testing via Burp Suite Pro
  • Review of HTTP security headers and SSL/TLS configuration
  • File upload functionality and user privilege separation testing

Key Findings

Outdated Plugin with Known CSRF Vulnerability User Enumeration via Login Error Messages Missing Security Headers (CSP, HSTS, X-Frame-Options) Weak Admin Password Policy Directory Listing Enabled on /uploads/ TLS 1.0 Enabled on Subdomain WordPress Version Disclosure

Outcomes

  • Delivered a full written report with executive summary and technical findings within 48 hours
  • Conducted a 1-hour debrief with the organization's volunteer IT coordinator
  • All Critical and High findings remediated by the client within 2 weeks
  • Performed complimentary re-test confirming successful remediation
Key Takeaway: A single unpatched WordPress plugin with a known CSRF vulnerability could have allowed an attacker to take over admin accounts, redirect donations, or exfiltrate donor PII. Nonprofits often lack dedicated security resources — even a basic annual assessment can close critical exposure gaps.

Scenario

In a controlled lab environment, simulated a full social engineering attack chain: spear-phishing email → credential harvest → post-authentication access → lateral movement. The purpose was to practice the full attack lifecycle and develop a reusable phishing simulation methodology for future client engagements.

Attack Chain

  • Performed OSINT on simulated "employee" profiles to craft believable lure emails
  • Stood up GoPhish infrastructure with a lookalike domain (registered and decommissioned within lab)
  • Crafted spear-phishing email mimicking an internal IT password reset request
  • 73% of simulated recipients clicked the link; 40% submitted credentials on the cloned portal
  • Used harvested credentials to authenticate to simulated VPN portal
  • From simulated VPN access, performed internal enumeration and identified next-hop targets

Key Observations & Lessons Documented

  • Urgency and authority cues in email subjects dramatically increased click rates
  • MFA would have broken the attack chain at credential harvesting stage
  • Lookalike domains that closely mirror internal naming conventions are highly convincing
  • Security awareness training that includes recognizing urgency tactics is most impactful

Methodology Output

This exercise produced a documented phishing simulation playbook now used as the foundation for Light Security's social engineering engagement methodology, including pre-campaign OSINT, lure construction guidelines, infrastructure setup procedures, and post-campaign reporting templates.

Key Takeaway: Phishing remains the #1 initial access vector in real-world breaches. Even technically sophisticated organizations with patched systems are vulnerable when the human element is not regularly tested and trained. MFA enrollment alone would have neutralized the post-phish access in this chain.
